# Contributing

## Workflow

1. Branch from `main`.
2. Keep changes focused and reviewable.
3. Run validation checks before opening a PR.
4. Include a clear description of why the change is needed.

## Pull Requests

- Add context, affected user/system impact, and test notes.
- Link related issues or docs when available.
- Ensure no secrets are committed.

## Operational Quality

- Keep incident/status logic changes small and explicit.
- Keep operational docs up to date with behavior changes.
- Preserve backwards compatibility for status consumers when possible.
