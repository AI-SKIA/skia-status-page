# Status Update Runbook

## Purpose

This runbook defines the operational workflow for publishing, superseding, and pruning status events in `incidents.json`.

It is the lifecycle companion to `docs/INCIDENT_EVENT_SCHEMA.md`.

## Authority and ownership

- **Primary owner:** Platform operations owner for SKIA status publishing.
- **Technical approvers:** Runtime/backend maintainer for benchmark and incident correctness.
- **Emergency override:** On-call incident commander during active production incidents.

No unauthorized actor should publish status events directly.

## Standard publish workflow

1. Gather source-of-truth evidence (incident report, eval output, capability release note).
2. Create/update event record in `incidents.json` following schema contract.
3. Validate JSON parse and required field integrity.
4. Confirm UI rendering on both:
   - `index.html` (summary panels)
   - `incidents.html` (timeline)
5. Commit and deploy with clear release note context.

## Adding a new event

For every new event:

- Add stable `id`.
- Set correct `type`.
- Set `status` and timestamp fields.
- Include `evidence`/`title` where meaningful.

Event should be inserted near the top unless a specific timeline reconstruction is needed.

## Superseding stale entries

Use supersession when older rows are no longer canonical but should remain auditable.

Required actions:

- Set `status: "superseded"`.
- Add `supersededReason` with explicit rationale.
- If applicable, link logically to the newer event by including the new ID in reason text.

Do not silently delete canonical historical records unless they are known placeholders/noise.

## Placeholder and benchmark hygiene rules

- `eval_result` with placeholder/non-provider score must not remain active.
- Allowed outcomes:
  - remove row (if purely placeholder noise), or
  - mark as `superseded` with reason and `providerBacked: false`.
- Active canonical baseline rows must be provider-backed.

## Pruning rules

Prune only when data volume harms readability/performance and history remains sufficiently represented.

Recommended policy:

- Keep all major `incident` records.
- Keep latest canonical `eval_result` per suite + significant milestone records.
- Supersede or remove repetitive transient placeholder rows generated by failed/incomplete runs.
- Never prune without an audit note in release context.

## Validation checklist (before publish)

- JSON valid.
- All new rows include stable ID and correct type.
- No active placeholder benchmark rows remain.
- `index.html` and `incidents.html` render without errors.
- Incident severity state on homepage reflects intended production posture.

## Rollback procedure

If bad data is published:

1. Revert to last known-good `incidents.json`.
2. Re-run validation checklist.
3. Re-deploy immediately.
4. Publish follow-up event if customer-visible confusion occurred.

## Operational cadence

- Incident-driven updates: immediate.
- Benchmark updates: after validated provider-backed eval run completion.
- Capability updates: on release/activation boundaries.
